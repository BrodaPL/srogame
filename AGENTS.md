# AGENTS.md

This file captures session context for collaborators and future AI agents.

## Project Summary
- Name: srogame
- Stack: Angular (standalone components), TypeScript
- Purpose: Browser game similar to OGame but simplified, turn-based (not real-time), supports single-player and small-scale multiplayer. PvE is primary, PvP is possible. Enemy AI will be simplistic and RNG-driven with scaling.
- Secondary purpose: Learning TypeScript and Angular through building the game.
- Persistence (current): Uses browser localStorage key `srogame:setup` and `srogame:player` (auth session). Galaxy is held in server memory and fetched via API. Accounts + sessions are stored in `server/data/auth.json`.
- Conventions: Use camelCase for variable names.
- Collaboration preference: Use planning mode by default for regular development tasks.
- Planning-question preference: When clarifying design decisions in planning mode, ask numbered questions with options and place the recommendation directly under each question (`Recommended: ...`) instead of collecting all recommendations in one block at the end.

## Current Behavior
- Route `/` shows main menu (load, singleplayer, multiplayer, encyclopedia, help/about).
- Route `/setup` shows setup form (game type, starting metal/crystal/deuterium, plus test-only checkboxes for random planets/starting ships) and requires login; player name is read-only from session.
- Route `/game` shows game view if config exists, otherwise prompts to go to setup. Shows a galaxy preview grid when in-memory galaxy exists.
- Route `/game/galactic` now shows the galaxy grid with own-fleet route overlays. Own active fleets are projected into the shared galaxy-presentation payload, rendered as green origin->target system arrows or dark-green returning target->origin arrows, aggregated by system route with count badges, and selected systems list own fleets currently stationed there. Cell value labels turn green/bold when the player has an active fleet currently present in one of their own systems.
- Route `/login` shows login/register form.
- Route `/encyclopedia` shows encyclopedia menu with ships/defences/buildings/technologies/mechanics links.
- Routes `/load`, `/multiplayer`, `/help` are placeholders.
- Route `/encyclopedia/ships` shows ship cards sourced from ship blueprints (images + stats + costs). Clicking a card image opens a popup that shows the optimized image immediately and then swaps to the matching `raw/` asset once it finishes loading.
- Route `/encyclopedia/buildings` shows building cards sourced from building blueprints (images + basic stats + costs). Clicking a card image opens the same large-image popup and loads the matching `raw/` asset on demand.
- Route `/encyclopedia/technologies` shows technology cards sourced from tech blueprints (images + basic stats + costs). Clicking a card image opens the same large-image popup and loads the matching `raw/` asset on demand.
- Route `/game/mission-planner` is now live for fleet launch. It supports `Move`, `Guard`, `Transport`, `Spy`, `Bombard`, `Siege`, `Recycle`, `Repair`, and `Colonize`, uses mission-first selection, origin/target coordinate resolution, right-side owned-planet rails via `MiniPlanetPreview`, ship-purpose filtering (`Military/Cargo/Utility/Carrier/Recycling`), compact cargo/travel panels, warning validation, round-trip fuel preview for non-spy missions, and fleet creation. Fleet Composition rows now split launch selection into `Ready` and `Damaged` counters, preserve damaged-hull selection through launch, show `Available: N (Damaged: D)`, and show an orange `No hyperspace drive.` tag for ships with `canJump === false`. `Bombard` and `Siege` now expose optional `Main` / `Secondary` / `Tertiary` bombard-priority dropdowns; empty priorities keep the old random targeting, while selected priorities persist on the fleet for later `Siege` ticks. `Move`, `Guard`, and `Transport` now also expose `Use Jump Gate` when the target has a known Jump Gate; approved Jump Gate launches always use exactly 1 travel turn, while foreign targets create a pending owner-approval request.
- Route `/game/operations` is now live and lists active fleets from the central game-state collection, including lifecycle state (`PENDING_JUMP_GATE`, `MOVING_TO_TARGET`, `IDLE`, `RETURNING`, failure-return states), current position, destination, ETA, ships, cargo, and fuel. Pending Jump Gate fleets stay parked at origin until the request resolves. Orbiting fleets with usable `ALLIANCE_DEPOT` support now expose a `Request maintenance` popup that can request fuel, `PLANETARY_BOMB`s, and `HullClass.SMALL` ships from the target planet.
- Route `/game/mail` is now live from the shared top menu as the communication center for `Messages` and `Requests`. It lists all player messages plus diplomacy requests, Jump Gate requests, and `ALLIANCE_DEPOT` maintenance requests, groups requests into `Pending` and `Resolved`, groups messages into `Unread` and `Read`, marks messages read on open, supports reply and new-message composition through a reusable popup, and allows single-player or whole-alliance sends (`alliance` currently means all human players at `ALLIED` status with the sender). Jump Gate requests support approve, reject, and cancel; maintenance requests support approve, partial approve with numeric per-item amounts, reject, and cancel. Resolved requests stay visible until manually deleted; pending incoming requests are the request count used by the End Turn blocker.
- Top menu communication behavior is now split: `Reports` shows only unread report count, `Mail` shows unread-message plus pending-incoming-request attention count, End Turn is blocked while unread mail or pending incoming requests exist, and when blocked a sticky shining Mail CTA is pinned near the top of the screen until the player opens Mail and clears the blocker.
- Route `/game/reports` is now live as a data-report inbox only. It excludes player-to-player mail, supports exact report-type tabs plus `All`, newest-first ordering, unread/read state, checkbox selection with delete, inline report detail view, `Preview location` for planet-linked reports via `MiniPlanetPreview`, separate scroll areas for inbox and report body, darker/compact inbox styling where read reports are grayed out, and a dedicated dossier-style detail layout for `EspionageReportData` with summary cards plus sectioned intel blocks.
- Route `/game/diplomacy` is now live from the shared top menu. It lists only discovered contacts (discovery = viewer has `lastReportData` for at least one planet currently owned by that player), excludes `(NEUTRAL && WAR)` contacts, shows known espionage-backed planets, creates treaty proposals, opens the shared compose-message popup for direct player mail, and shows an active-proposals summary that now points players to Mail for accept/reject/cancel management. Treaty proposals are limited to one outgoing proposal total per turn, block duplicate pending pair proposals, apply immediately on acceptance, can be cancelled by the proposer until resolved, and auto-expire on the proposer's next turn if unanswered. `PASSIVE` remains visible in stored relations but is intentionally not proposeable in the player UI yet.
- Route `/game/imperium` now includes a per-planet abandon flow. A planet can be abandoned only if it is not the player’s last owned planet. Abandoning immediately removes it from the player, assigns a fresh neutral owner, sets a `PASSIVE` relation between the old owner and that new neutral, cancels all building/shipyard/research queues plus research-helper links on the planet, keeps local `rBDSFTQ` resources/ships/defences in place, leaves active fleets untouched, and preserves immediate intel visibility for the old owner.
- Tutorials are now in phase-2 desktop polish for all currently supported tutorial views: `Galaxy View`, `Planet View`, `Mission Planner`, `Reports View`, `Operations View`, `Imperium View`, `Buildings View`, `Production View`, and `Researches View`. `TutorialService` supports target-aware steps via `data-tutorial-id`, staged `focus -> highlight -> bubble` flow, spotlight dimming, scroll locking while open, target-aware desktop dock placement, and per-view step preparation hooks so views can reveal hidden targets safely before spotlight measurement. Whole-view intro steps are now intentionally allowed to omit `targetId`, which means step 1 can be a bubble-only orientation step without spotlighting one specific control. `Galaxy View` auto-selects and preloads the home system whenever it opens with no current selection, and its live tutorial has 6 targeted steps (grid, preselected home system, grid meaning, preview panel, planet mini-cards, notes). `Planet View` now has a whole-view intro plus 5 targeted steps (overview/warnings, planet navigation, tabs, representative resource building card, queues) and safely auto-switches between `Resources` and `Queues` tabs during tutorial flow without synthetic gameplay actions. `Mission Planner` now preselects a default owned target when none is prefilled and has a whole-view intro plus 6 targeted steps (target card, origin rail, mission details, travel summary, fleet composition, launch readiness). `Reports View` now preselects the first visible report into the detail pane without marking it read and has a whole-view intro plus 5 targeted steps (bulk actions, inbox list, report header, detail stats, full report body). `Operations View` now exposes a main-state target and a primary fleet-card target, and its tutorial uses a whole-view intro before the live-data steps; it auto-opens only when at least one active fleet exists so the walkthrough lands on meaningful live data. `Imperium View` now uses a whole-view intro before empire totals, attention, owned-planets, representative planet card, fleet totals, and building statistics. `Buildings View` now uses a whole-view intro before category toggle, building queue card, compact list, representative building row, and owned-planets rail. `Production View` now uses a whole-view intro before mode toggle, queue grid, shipyard queue card, compact production list, representative ship row, and owned-planets rail. `Researches View` now exposes anchors for the labs panel, queue area, technology list, and representative technology card, and its tutorial auto-open is deferred until after the first render so the walkthrough appears immediately instead of only after later UI interaction. Current desktop dock behavior is VN-style vertical stacking: bubble above/below the character while the whole dock sits to the chosen side of the highlighted target. Mobile-specific tutorial layouts are not planned yet.
- Basic diplomacy is now live in the domain/server layer with symmetric `SELF` / `ALLIED` / `PEACE` / `PASSIVE` / `WAR` relations stored on `Galaxy`. Admin/test-only endpoints can read/write diplomacy, mission validation now respects friendly targets, `Move` can idle in allied/peace orbit, `Transport` can deliver to allied/peace planets and return, and orbit encounters merge allied defenders while `PEACE` prevents auto-combat. `PASSIVE` is currently stored and transported as future-facing scaffolding; its special plunder behavior is not implemented yet.
- `Colonize` was explicitly checked during the abandon-planet work and was found to still use the generic placeholder mission path. It now has a dedicated mission implementation. It can still claim unowned planets, and it can also reclaim `PASSIVE` neutral planets such as newly abandoned worlds; those recolonizations merge the arriving fleet/cargo into the planet, transfer ownership immediately on arrival resolution, and remove the temporary neutral owner when that owner no longer holds any planets.
- Fleet and planet ship storage now use `ManyShips` (`undamagedShipsCount` plus per-ship damaged hull entries) instead of keeping all healthy ships expanded as `ShipInstance[]`. Hull damage persists after battles, shields reset after battle, and damaged ships still count as fully usable for mission selection for now.
- Route `/game/planet` shipyard tab now includes a `Ship Damage Status` panel. It always renders, shows `No ships available.` when empty, and otherwise displays undamaged/damaged ship percentages with color-coded health state plus tooltips listing undamaged stacks and damaged-ship missing-HP summaries.
- Route `/game/planet` now also exposes a `Trade Port` button once `INTERSTELLAR_TRADE_PORT` is built. It opens a local popup that lists that planet current turn trade offers, shows `Get ... / Cost ... (modifier)` labels, and lets the player spend local resources instantly on one-use-per-turn offers.
- End Turn now includes automatic ship repair. Planet ships are repaired first, then owned/allied idle orbit fleets (`IDLE` and `MISSION_FAILURE_IDLE`). Ship repair uses current shipyard power, non-drone `REPAIR_EQUIPMENT`, and repair drones; no extra resource or energy cost is applied yet.
- End Turn mission logic now also supports mission-owned idle orbit processing. `Siege` keeps bombarding while hostile, but now stores fleet bombard priorities, applies an extra `50%` per-shot trigger failure during its bombardment phase, and consumes additional fuel reserve each orbit turn (`fuelCost / travelTurns`, rounded up); if only return fuel remains, the siege stops immediately and returns. `Repair` keeps its non-hostile orbit constraints, and `Recycle` can remain in `IDLE` orbit to harvest `spaceDebris` each turn with `RECYCLE_EQUIPMENT` until cargo is full or debris is exhausted, then auto-return.
- Orbiting non-hostile fleets now distinguish between passive orbit and active guard orbit. `Move` to a non-hostile foreign/unowned planet now resolves into `HOLD + PASSIVE_HOLD`, which does not defend the planet owner against direct assault but will intercept hostile orbit-staying missions. The player-facing `Guard` mission uses the existing internal `DEFEND` mission type and resolves into `DEFEND + GUARDING`, joining the target planet owner’s coalition on `SELF` / `ALLIED` / `PEACE` / unowned targets. If diplomacy later becomes hostile, guarding fleets downgrade to passive orbit instead of auto-returning.
- Shared repair info is now visible in the powers panel, Planet View, Imperium, Operations, and Mission Planner. Current compact labels are `Ship repair`, `Industry repair`, `Drone repair`, plus mission/fleet labels `Ship repair capability` and `Drone repair capability`. Repair-related warnings currently focus on damaged ships present vs missing repair capability.
- Production queue management is now live for buildings and shipyard production. Buildings queue supports drag-and-drop reorder plus cancel; shipyard uses one shared mixed queue for ships + defences in queue-management UIs so visible order matches real execution order. Reordering preserves invested progress. Cancel rules are: `0 progress -> 100% refund`; started building entries refund `75%` of their queued cost; started shipyard stacks deliver already completed units to the planet first and refund `75%` of the unfinished remainder. Current research queue still has no cancel/reorder management.
- Home systems now get one guaranteed low-level neutral neighbor planet when `neutralBotsAmount > 0`. This applies during galaxy creation for current human starting players: the generator reuses level-3 RNG buildings/tech/ships/resources, prefers an existing extra non-asteroid planet in the same system, and creates one if the home system has no secondary non-asteroid planet.
- Space battles now generate simple accumulating debris fields on the target planet. `Planet.rBDSFTQ.spaceDebris` gains a random salvage share of destroyed-ship value after battle: metal/crystal keep `20%..30%`, deuterium keeps `5%..10%`, and attacker cargo is also added to the destroyed-value pool when the attacking fleet is wiped out.
- Terraformer now has split mechanics. Completed terraformer levels permanently increase planet size using the terraformer blueprint `production1` value at the highest completed level reached, independent of later damage or energy loss. Separately, current terraformer effectiveness reduces only negative `metal/crystal/deuterium/industry/research` planetary-parameter penalties by `1%` per effective level, capped at `1.0`; it does not affect `anomaliesAndNoise` or `hyperspaceParameters`.
- Interstellar Trade Port is now live as a local exchange mechanic. Offer count equals Trade Port level, offers refresh on every global turn, each offer can be used once per turn, and offers stay fixed for the turn once rolled. Offer cap uses current effective Trade Port capacity, which now scales with current Trade Port effectiveness, planet `hyperspaceParameters`, `HYPERSPACE_TECHNOLOGY`, `GRAVITON_TECHNOLOGY`, and `+20%` per built `JUMP_GATE` level. Resource valuation is `3 metal = 2 crystal = 1 deuterium`, amount steps are `20%..100%` of cap in `20%` increments, and total offer cost uses the rolled `5%..40%` modifier minus `1%` per Trade Port level and minus `1%` per Jump Gate level. The cost modifier can go negative, but final offer cost is floored at `0`.
- SENSOR_PHALANX backend phase-1 is now live without a dedicated view yet. `GET /api/game/sensor-phalanx/capabilities` and `POST /api/game/sensor-phalanx/scan` expose origin-planet capabilities plus active scans. Range uses `production1`, `anomaliesAndNoise`, and live building effectiveness; active scan range is `max(1, floor(normalRange / 2))`; scan cost is `production2` deuterium; scans per turn are `floor(sqrt(level) * effectiveness)`. Passive detection runs at turn start after other turn calculations, checks incoming fleets whose target planet is within normal range, and creates text `SENSOR_PHALANX_REPORT`s only for newly visible contacts. Active scans reveal only `direction`, `fleetSize`, `etaTurns`, and `isAllied`.

### How to use skills
- Discovery: The list above is the skills available in this session (name + description + file path). Skill bodies live on disk at the listed paths.
- Trigger rules: If the user names a skill (with `$SkillName` or plain text) OR the task clearly matches a skill's description shown above, you must use that skill for that turn. Multiple mentions mean use them all. Do not carry skills across turns unless re-mentioned.
- Missing/blocked: If a named skill isn't in the list or the path can't be read, say so briefly and continue with the best fallback.
- How to use a skill (progressive disclosure):
  1) After deciding to use a skill, open its `SKILL.md`. Read only enough to follow the workflow.
  2) When `SKILL.md` references relative paths (e.g., `scripts/foo.py`), resolve them relative to the skill directory listed above first, and only consider other paths if needed.
  3) If `SKILL.md` points to extra folders such as `references/`, load only the specific files needed for the request; don't bulk-load everything.
  4) If `scripts/` exist, prefer running or patching them instead of retyping large code blocks.
  5) If `assets/` or templates exist, reuse them instead of recreating from scratch.
- Coordination and sequencing:
  - If multiple skills apply, choose the minimal set that covers the request and state the order you'll use them.
  - Announce which skill(s) you're using and why (one short line). If you skip an obvious skill, say why.
- Context hygiene:
  - Keep context small: summarize long sections instead of pasting them; only load extra files when needed.
  - Avoid deep reference-chasing: prefer opening only files directly linked from `SKILL.md` unless you're blocked.
  - When variants exist (frameworks, providers, domains), pick only the relevant reference file(s) and note that choice.
- Safety and fallback: If a skill can't be applied cleanly (missing files, unclear instructions), state the issue, pick the next-best approach, and continue.
